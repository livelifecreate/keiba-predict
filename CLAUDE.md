## 実行コマンド

```bash
python3 saturday_predict.py          # 土曜日の全レース予想（2勝クラス以上）
python3 sunday_predict.py            # 日曜日の全レース予想（2勝クラス以上）
python3 weekend_predict.py --sat     # 土曜（直接実行）
python3 weekend_predict.py --sun     # 日曜（直接実行）
python3 verify/bet_metrics_standard.py              # 全期間ROI検証（9馬券種×クラス別）

```

## トラブルシューティング

### JRAオッズ取得失敗時

予想実行中に以下が表示された場合:
```
⚠ [JRA] パラメータエラー検出 — チェックサム式が変更された可能性があります
        今すぐ修正: python3 tools/jra_checksum_diag.py --fix
```

**対処手順:**
1. `python3 tools/jra_checksum_diag.py` を実行して診断
2. 不一致が出たら `python3 tools/jra_checksum_diag.py --fix` で自動修正
3. 修正後に `python3 saturday_predict.py` を再実行

**注意:** JRAオッズは発走前のみ取得可能。発走後の `[オッズ] 取得失敗` は正常動作。
パラメータエラーはチェックサム式変更が原因のため、上記コマンドで解決できる。

## オッズ取得ルール（2026-06-27更新）

### 状況別の取得元（必ず区別すること）

| 状況 | データソース | キー/関数 |
|---|---|---|
| **バックテスト（過去データ）** | `cache/race_result/*.json` の `entries[].odds` / `entries[].popularity` | キャッシュ直読み・外部取得禁止 |
| **当日予想 オッズ（第1優先）** | JRA公式出馬表 `pw01dde01...` → `HorseEntry.odds` / `HorseEntry.popularity` | `_fetch_jra_odds()` |
| **当日予想 オッズ（フォールバック）** | netkeiba `shutuba.html` → `Shutuba_Table` | `fetch_odds()` |
| **当日予想 馬場状態** | JRA公式出馬表 `accessD.html` → `parse_race_info().track_condition` | `_fetch_jra_odds()` の戻り値に含まれる |

### バックテスト時のオッズ使い方（外部アクセス不要）

```python
for e in data["entries"]:
    odds = e.get("odds", 0)        # 単勝オッズ
    pop  = e.get("popularity", 99) # 人気
```

### 当日予想のオッズ・馬場状態取得フロー（weekend_predict.py）

```python
jra_odds_map, jra_track = _fetch_jra_odds(race_id, race_date, entries)
if jra_odds_map:
    odds_map = jra_odds_map          # JRA優先
else:
    odds_raw = fetch_odds(race_id)   # netkeibaフォールバック
```

- **馬場状態は出馬表（accessD.html）から取得**: `parse_race_info().track_condition` に当日の最新値が入っている
- **`/keiba/baba/` は使わない**: このページは「前日正午現在」の古い情報であり、当日の実際の馬場状態と異なる場合がある
- **発走前**: JRA公式出馬表からオッズ・馬場状態を同時取得できる
- **発走後（当日中）**: オッズは `0.0`（発走後は数値なし）→ 正常挙動
- netkeiba `odds_get_form.html` は全オッズが `---.-` (JSプレースホルダー) → 実質使えない

### JRA URL生成（jra_scraper.py）の注意点

チェックサム計算式（2026-06-27更新・全18データポイントで検証済み）:
```python
race_contrib = ((race - 1) * 181 + (64 if race >= 10 else 0)) % 256
base = (20 + venue*157 + kai*210 + nichi*48) % 256
checksum = (base + race_contrib) % 256
```
- race 1-9: (race-1) * 181（mod 256）
- race 10-12: さらに+64オフセット
- venue係数: 157（旧10は廃止）、kai係数: 210（旧84は廃止）、nichi係数: 48（不変）
- JRA出馬表は**発走前のみ**アクセス可能。発走後はパラメータエラーになる（正常挙動）

## 事前確認エージェントルール（2026-06-20制定）

**予想・バックテスト結果を出力する前に、以下を必ず自己チェックする:**

### 予想時チェックリスト

1. **オッズ取得確認**: `odds_map` が空でないか
   - `_fetch_jra_odds()` (JRA公式) が第1優先 → 発走前のみ有効
   - フォールバック: `fetch_odds()` (netkeiba) → 実質`---.-`で機能しないことが多い
   - 発走後レースのオッズ未取得は正常。⚠警告を出してスキップ
2. **馬場状態取得確認**: `jra_track` が取れているか → `_fetch_jra_odds()` の第2戻り値
   - 稍重/重/不良なら道悪実績スコアが加算される → 馬場確認は必須
3. **クラス判定確認**: `race_class` が正しく判定されているか（OPと3勝とGIIIの誤判定に注意）
4. **見送りルール適用確認**: 18頭/乖離3〜5pt/2勝以下/重賞が正しく除外されているか
5. **馬名マッチング確認**: netkeiba馬名とオッズマップの馬名が一致しているか

### バックテスト時チェックリスト

1. **オッズ取得元の確認**: バックテストは必ず `cache/race_result/*.json` の `entries[].odds` を使う
   - JRA公式・netkeibaへの外部アクセスは不要・禁止（過去レースは発走後でURL無効のため）
2. **対象期間・クラス・馬場の確認**: コードに明示してから実行
3. **投資額の計算基準**: 1点=100円固定。複数点の場合は `N * pts * 100` で投資額計算
4. **的中判定ロジック**: `frozenset` の比較で馬番一致を確認（馬名ではなく馬番）
5. **払戻データ存在確認**: `cache/payouts/` が存在しないレースはスキップしているか
6. **クラス分類の表示**: 全体・クラス別・芝/ダート別を必ず出力

### コードミスが多いパターン（要注意）

- `'3連複'` vs `'三連複'` → 正しいキーは **`'3連複'`**
- 馬番パース `_pn_for_count()` を使わず greedy パースすると誤読 → 必ず件数を渡す
- `cache_get_before()` の `max_days_back` デフォルト90 → バックテストで使用
- 芝/ダート混在でコメントする際は「芝のみ実運用対象」を前提にすること

## 見送りロジック バックテスト結果・改定（2026-06-16）

**バックテスト対象:** 401R（in-memory採点・2勝クラス以上）

**廃止した見送りルール:**
- 断然人気（軸馬オッズ<2倍）の見送り → **廃止**（5BOX回収率141%・複勝率87.9%で明確に黒字）
- 乖離≥5pt 高信頼7点推奨 → **廃止**（単勝ROI50%・5BOX ROI40%でバックテスト不合格）

**維持した見送りルール（バックテストで正当化）:**

- 乖離3〜5pt: 5BOX ROI69% → 見送り維持
- 2勝クラス以下: 単勝ROI79% / 5BOX ROI73% → 見送り維持
- 重賞（GI/GII/GIII）: 三連複ROI6%以下・n=92 → 見送り確定
